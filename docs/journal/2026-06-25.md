# 2026-06-25

**Goal:** Close the Phase D SIEM hop — get Suricata's EVE alerts off pfSense and into
the Wazuh dashboard, end to end.

**Did:**
- **Abandoned the syslog path** carried over from last session. Confirmed it is a dead
  end, not a misconfig: pfSense emits Suricata EVE under the FreeBSD `LOCAL1` facility,
  but pfSense remote-logging only relays its own managed streams and never forwards
  `LOCAL1`. tcpdump on Wazuh showed zero packets despite valid config + successful ping.
- **Decided against the standard "Wazuh agent on the firewall" pattern.** Installing an
  unsupported package on pfSense (a FreeBSD production router) means enabling the upstream
  FreeBSD pkg repo — the most common way to corrupt a pfSense box — and a pfSense update
  can wipe or break the agent. The blast radius of a firewall failure (whole-house
  outage) outweighs the convenience. Decision recorded as ADR-0007. **Nothing unsupported
  is ever installed on pfSense.**
- **Built a pull-based pipeline instead** (firewall stays untouched):
  - Enabled SSH on pfSense (System → Advanced → Admin Access → Secure Shell).
  - Generated a dedicated ed25519 keypair on the Wazuh box (`/root/.ssh/suricata_pull`,
    comment `wazuh-eve-pull`, no passphrase). Private key never leaves Wazuh.
  - Installed the public key on the pfSense `admin` user **locked with a forced command**
    so it can do exactly one thing — `cat` the EVE file — and nothing else:
    `command="/bin/cat /var/log/suricata/suricata_vtnet147403/eve.json",restrict ssh-ed25519 …`.
    No shell, no port-forwarding, no agent forwarding.
  - Wazuh reaches pfSense at its own MONITORING gateway `10.10.40.1` (MONITORING→any is
    `Pass`). SERVICES→MONITORING on 22 stays blocked by default-deny — confirmed during
    setup — so the pull uses the allowed outbound direction.
  - Wrote `/usr/local/bin/pull-eve.sh` (`0700`, root): pulls the full `eve.json` over the
    forced-command key, compares its line count to a saved offset
    (`/var/log/suricata-pfsense/.lastcount`), appends **only new lines** to a local copy
    at `/var/log/suricata-pfsense/eve.json`, and resets/re-reads from the top if the
    remote file shrinks (log rotation). Dedup proven: two back-to-back runs both reported
    30,477 lines — no doubling.
  - Scheduled it every minute (`* * * * * /usr/local/bin/pull-eve.sh`).
  - Pointed Wazuh at the local file with an `ossec.conf` `localfile` block
    (`<log_format>json</log_format>`, `<location>/var/log/suricata-pfsense/eve.json</location>`)
    and **removed the dead `<remote>` syslog-listener block** from the abandoned attempt
    in the same pass.

**Broke / fixed:**
- Long debugging arc that ended in a collector gotcha: **Wazuh's log collector tails from
  end-of-file**, so the pre-existing alert backlog was skipped. Nothing showed until a
  manager restart plus freshly-appended alerts flowed through.
- noVNC console cannot copy text and mangled pasted quotes/slashes. Transcribing the SSH
  public key by eye introduced a wrong character → `Permission denied (publickey)`. Fixed
  by routing the key through a real terminal: Wazuh `scp`'d the `.pub` to the Proxmox host
  (an allowed outbound direction), where copy/paste is reliable. The `echo | crontab -`
  install also failed to the same quote-mangling — fixed by writing the cron line to a
  file with single quotes, then `crontab <file>`.
- Used the wrong validator first: `wazuh-logtest -t` is **not** a config check (that flag
  doesn't exist). `wazuh-analysisd -t` is the correct offline validator — ran it clean
  before restarting.

**End-to-end validation:**
- `wazuh-logtest` on a real alert line confirmed Phase 2 decoding (Suricata decoder) and
  Phase 3 rule match — rule **86601** ("Suricata: Alert - SURICATA STREAM …"), level 3,
  groups `ids,suricata`, "Alert to be generated."
- Wazuh dashboard (`rule.groups:suricata`, Threat Hunting) shows **1,783 Suricata alerts**
  indexed and charted over time, sourced from the `bgranilloserver` agent.

**Honest characterization:** all 1,783 are `SURICATA STREAM` TCP-anomaly events from the
stream engine — a large share self-generated by the SSH pull traffic itself. This proves
the full IDS→SIEM pipeline works end to end (the Phase D objective). It is **not** yet
detection of real attack signatures; those (Nmap, Hydra, Metasploit, SQLi, C2 callback)
are Phase E, where ETOpen rule hits will stand out against this STREAM baseline.

**Housekeeping:** reverted the Wazuh manager `<logall>` from `yes` (left on for
diagnostics) back to `no` and restarted. STREAM-noise suppression deliberately **deferred**
to Phase D-polish / Phase E — the correct fix is source-scoped per-SID suppression (silence
these SIDs only from the puller IP), not a blanket mute that would create a blind spot.

**Learned:** decoupling log shipping from the appliance keeps the firewall upgrade-safe and
minimizes blast radius; a forced-command SSH key is least-privilege done right (one command,
nothing else); the Wazuh collector reads new appends only, so a restart is needed to pick up
an existing backlog.
