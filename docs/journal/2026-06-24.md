# 2026-06-24

**Goal:** Phase D — deploy Suricata IDS on pfSense and forward alerts to Wazuh.

**Did:**
- Snapshotted pfSense VM 200 (`pre-suricata`) before any change.
- Installed Suricata `7.0.8_5` via System → Package Manager.
- Global Settings: enabled ETOpen Emerging Threats + Feodo Tracker Botnet C2 IP +
  ABUSE.ch SSL Blacklist rulesets; update interval 1 Day; Live Rule Swap OFF
  (avoids a RAM spike on the 16 GB host); GeoLite2 OFF. Downloaded rules — all
  three sources fresh (result "success").
- Created a Suricata interface on LAN (vtnet1) in IDS mode: Block Offenders OFF
  (IDS-only; also dodges the netmap/inline-IPS crash on virtualized NICs),
  Run Mode AutoFP, Detect-Engine Profile Medium, EVE JSON ON.
- Enabled attack-mapped rule categories: `emerging-scan`, `emerging-exploit`,
  `emerging-exploit_kit`, `emerging-shellcode`, `emerging-attack_response`,
  `emerging-sql`, `emerging-web_server`, `emerging-web_specific_apps`,
  `emerging-malware`, `emerging-user_agents`, plus Feodo Tracker and ABUSE.ch
  SSL Blacklist. (Deliberately left `emerging-dns` off — too noisy on a Pi-hole network.)
- Started Suricata on LAN — came up green.
- **Confirmed first live detection.** From the Proxmox host:
  `pct exec 100 -- curl -s http://testmynids.org/uid/index.html` (the canonical
  NIDS smoke test) tripped **"GPL ATTACK_RESPONSE id check returned root", SID
  `1:2100498`**, Class "Potentially Bad Traffic", Priority 2, src
  `65.8.20.126:80` → dst `10.10.10.100` (Pi-hole), in Services → Suricata → Alerts.
  Action was an alert, not a block — confirming IDS mode.
- Attempted SIEM forwarding via syslog (chosen specifically to avoid installing an
  unsupported Wazuh agent on the FreeBSD firewall): set the Suricata interface
  EVE Output → Syslog (facility LOCAL1); enabled pfSense remote logging to
  `10.10.40.10:514`; added a UDP-514 syslog listener to Wazuh `ossec.conf`
  (allowed-ips `10.10.40.1`). Confirmed the manager is listening
  (`ss -lunp` shows `wazuh-remoted` on UDP `0.0.0.0:514`) and pfSense → Wazuh
  ping is 0.0% loss.

**Broke / fixed:**
- `wazuh-manager` failed to start twice from console-paste-mangled `ossec.conf` XML
  (a `)` where a `/` belonged in a closing tag; later a duplicate `<remote>` block)
  → retyped in `nano` and validated the whole file, not just the tail. Service active.
- SSH from SERVICES (10.10.10.x) → MONITORING (10.10.40.10) on port 22 is blocked by
  default-deny (correct — only 1514-1515 and dashboard 443 are allowed). Reached
  VM 201 via the Proxmox console instead.

**Learned:** pfSense's Suricata EVE-to-syslog output is emitted under the FreeBSD
syslogd **LOCAL1** facility, but pfSense remote-logging only forwards its own managed
log streams (firewall, system, DHCP, etc.) — it does **not** forward LOCAL1. So the EVE
messages are written locally on pfSense and never leave the firewall. `tcpdump -n -i any
udp port 514` on Wazuh showed **zero packets** while the curl was fired repeatedly,
despite a valid config and a successful ping. This is an architectural gap in how the two
pfSense features wire together, not a misconfiguration.

`testmynids` curl and confirm the alert in the Wazuh dashboard end-to-end. Revert the
temporary Wazuh `<logall>yes</logall>` back to `no` to avoid disk growth.
